\begin{abstract}
PCOS Pathfinder is a data-grounded clinical decision-support prototype for earlier identification of Polycystic Ovary Syndrome (PCOS) and broader differential workup in women's health. The system uses the required PCOS clinical dataset as its modelling foundation, adds a synthetic endometriosis-overlap prompt for differential reasoning, and uses a lightweight single-cell gene-list check as biological context. The final implementation consists of six reproducible notebooks, three saved model artifacts, a Streamlit prototype, exported metrics and figures, and this report. The project is explicitly framed as decision support, not standalone diagnosis.
\end{abstract}

\section{Challenge Context}

The Biohackathon 2026 challenge asks participants to improve diagnostic accuracy for women's health conditions using available datasets and a practical implementation plan. PCOS is used as the core case because it is common, heterogeneous, and often diagnosed late. A useful solution must therefore do more than predict a binary label: it should help clinicians identify high-risk patients earlier, explain the drivers of risk, and avoid tunnel vision when related conditions have overlapping presentations.

\subsection{Project Goal}

The project goal is to build \textbf{PCOS Pathfinder}, a clinician-facing decision-support workflow with three linked components:

\begin{enumerate}
  \item \textbf{Frontline screening model}: a high-recall PCOS risk flag using symptoms and basic vitals only.
  \item \textbf{Enhanced diagnostic-support model}: a stronger model using labs and ultrasound features.
  \item \textbf{Endometriosis-overlap prompt}: a synthetic-data workflow nudge for broader differential workup.
\end{enumerate}

\subsection{Supplementary Endometriosis Dataset}

The supplementary endometriosis dataset contains 10,000 synthetic rows and seven variables: age, menstrual irregularity, chronic pain level, hormone-level abnormality, infertility, BMI, and diagnosis. It is used only for a differential-diagnosis prompt. Because the data is synthetic, the resulting model is not presented as clinically validated.

\subsection{Single-Cell Datasets}

The provided single-cell archives contain 10x-style outputs with \code{barcodes.tsv.gz}, \code{features.tsv.gz}, and \code{matrix.mtx.gz}. Notebook 06 performs a limited gene-list peek, not full single-cell analysis. It inventories 20 PCOS single-cell samples and 2 endometrium single-cell samples, then checks whether literature-supported PCOS genes appear in a sample feature list.

\section{Data Cleaning and Quality Controls}

Data cleaning is implemented in the notebook setup cells generated from \code{scripts/create\_training\_notebooks.mjs}. The key steps are:

\begin{itemize}
  \item Load the \code{Full\_new} sheet.
  \item Standardize names to lowercase snake case.
  \item Coerce numeric fields with logging of non-numeric cells.
  \item Drop identifiers: \code{sl\_no}, \code{patient\_file\_no}.
  \item Drop \code{blood\_group}, because the encoded values 11--18 are not ordinal.
  \item Drop \code{marriage\_status\_yrs}, because it is clinically sensitive and not a defensible diagnostic signal.
  \item Engineer \code{cycle\_irregular\_flag} from the source \code{cycle\_r\_i} coding.
  \item Cap visibly impossible or model-dominating outliers in FSH, LH, FSH/LH, vitamin D3, and progesterone.
  \item Use median imputation inside each scikit-learn pipeline, so imputation is saved with the model artifact.
\end{itemize}

\subsection{Important Data Caveat}

The source column \code{Cycle length(days)} is labelled ambiguously. The observed range is 0--12 with a median of 5, so it is treated in the app as \emph{duration of menses bleeding in days}, not the interval between cycles. The Streamlit input is correspondingly labelled \emph{Duration of menses bleeding (days)} and constrained to 0--12 days. This avoids out-of-distribution predictions from a default value such as 28.

\subsection{Candidate Models}

Both PCOS modelling notebooks compare a dummy baseline, logistic regression, random forest, and gradient boosting. The best model is selected by cross-validated performance on the training split. Thresholds are selected without using the held-out test set.

\section{Validation Strategy}

A single stratified train/test split is used with \code{test\_size = 0.25} and \code{random\_state = 42}. Model selection uses five-fold stratified cross-validation on the training split. For each chosen model, action thresholds are selected using out-of-fold training probabilities from \code{cross\_val\_predict(method="predict\_proba")}. The threshold sweep targets recall at or above 0.90 while maximizing specificity, precision, and F1 in that order.

The held-out test set is then evaluated once. This avoids the earlier risk of choosing a threshold on the test set and reporting optimistic metrics.

\section{Explainability}

Explainability is treated as a core product feature. Global drivers are exported as feature-importance tables, while patient-level explanations are generated with SHAP. For tree models, the workflow uses \code{shap.TreeExplainer}. For logistic regression, the model artifacts store a 100-row transformed training background for \code{shap.LinearExplainer}.

\subsection{Global Drivers}

For the screening model, the top global drivers are skin darkening, hair growth, weight gain, BMI, menses duration, age, and random blood sugar. For the enhanced model, follicle counts dominate, followed by skin darkening, weight gain, hair growth, AMH, menses duration, BMI, and cycle regularity.

\section{Differential-Diagnosis Prompt}

The endometriosis-overlap model is intentionally modest. It is trained on the supplementary synthetic dataset and is used only to prompt broader clinical reasoning. It should not be described as a validated endometriosis diagnostic model. In the prototype, a high overlap score displays a warning to consider broader gynecologic differential workup, especially when chronic pain, infertility, hormone-level abnormality, BMI, and menstrual irregularity indicate symptom overlap.

\section{Prototype}

The Streamlit prototype is implemented at \code{src/app.py}. It loads the three joblib model artifacts and renders:

\begin{itemize}
  \item \textbf{Patient assessment}: intake form, screening result, enhanced result, and endometriosis-overlap prompt.
  \item \textbf{Population view}: source-cohort symptom prevalence and headline model metrics.
  \item \textbf{About}: plain-language model descriptions, threshold strategy, and caveats.
\end{itemize}

Risk tiers are anchored to each model's action threshold so that tier labels cannot contradict binary action flags:

\begin{itemize}
  \item Low: probability below threshold.
  \item Moderate: probability from threshold to threshold plus 0.20, capped at 0.90.
  \item High: probability above the moderate band.
\end{itemize}

\section{Biological Rationale from Single-Cell Feature Lists}

Notebook 06 extracts feature symbols from one PCOS single-cell sample and confirms the presence of a curated list of PCOS-relevant genes, including genes linked to steroidogenesis, androgen signalling, follicular development, AMH signalling, insulin signalling, and PCOS GWAS loci. This is used only as biological context. No single-cell quality control, normalization, integration, clustering, or differential expression analysis is claimed.

\section{Limitations and Risk Controls}

\begin{itemize}
  \item \textbf{Small PCOS dataset}: 541 patients is not enough for deployment-level validation.
  \item \textbf{Single-source geography}: the data comes from hospitals in Kerala, India and may not generalize globally.
  \item \textbf{Synthetic endometriosis data}: the differential module is a workflow prompt only.
  \item \textbf{Limited equity variables}: race, income, geography, and access-to-care measures are unavailable.
  \item \textbf{Clinical safety}: all outputs are framed as risk, triage, and decision support, not diagnosis.
  \item \textbf{Version sensitivity}: scikit-learn model artifacts are sensitive to version skew, so dependencies are pinned.
\end{itemize}

\section{Conclusion}

PCOS Pathfinder satisfies the challenge by combining a required-dataset modelling core, an explainable tiered clinical workflow, a differential-diagnosis prompt, and a realistic implementation path. The most defensible pitch is not that the system diagnoses PCOS automatically, but that it helps clinicians identify high-risk patients earlier, choose the next appropriate investigation, and avoid missing overlapping women's health conditions.
